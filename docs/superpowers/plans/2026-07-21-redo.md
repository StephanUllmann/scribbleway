# Redo (Ctrl+Shift+Z) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a non-destructive redo path so undoing a shape mutation can be reversed with `Ctrl+Shift+Z`, a D-Bus/`AppletBackend` slot, and a Plasma applet button — without changing existing undo semantics.

**Architecture:** Keep the existing linear undo stack (`m_history`) and add a parallel redo stack (`m_redo`). On every real mutation snapshot, clear redo. On `undo()`, push the live shape list onto redo before restoring history. On `redo()`, push the live list onto history before restoring redo. Reuse the existing `m_isApplyingUndo` flag (or rename to `m_isApplyingHistory`) so `setShapes()` does not re-snapshot while applying either direction. Wire `OverlayController::redo()`, local shortcut `action_redo` (`Ctrl+Shift+Z`), QML `Shortcut`, `AppletBackend::redo()`, and an applet "Redo" button beside Undo.

**Tech Stack:** Qt 6 / C++20, QAbstractListModel, QML Shortcuts, D-Bus (`QDBusConnection::ExportAllSlots`), Plasma applet QML.

## Global Constraints

* **Destructive-new-edit rule:** Any new history snapshot from a user mutation MUST clear the redo stack (standard editor behavior).
* **Undo compatibility:** Existing undo tests and behavior MUST keep passing; undo remains pop-from-history + restore.
* **History cap:** Both stacks stay capped at 50 snapshots (same as today's `m_history`).
* **No canUndo/canRedo UI required:** Match current Undo button — always clickable; no-ops when the stack is empty (YAGNI).
* **Shortcut default:** `action_redo` defaults to `Ctrl+Shift+Z` and is a *local* overlay shortcut (same family as `action_undo`), not a KGlobalAccel global.
* **Selection after redo:** Mirror `OverlayController::undo()` — clear `m_selectedIndex` to `-1` and notify selection/shapes.
* **Excalidraw impact:** None. Redo only restores in-memory `QList<QVariantMap>` snapshots; no schema/key changes.

Replace `saveHistorySnapshot()` (lines 99–105) with:
```cpp
void ShapesModel::saveHistorySnapshot()
{
    m_history.append(m_shapes);
    if (m_history.size() > 50) {
        m_history.removeFirst();
    }
    // New user mutation invalidates the redo branch.
    m_redo.clear();
}
```

Replace `undo()` (lines 107–116) with:
```cpp
void ShapesModel::undo()
{
    if (m_history.isEmpty()) {
        return;
    }
    m_isApplyingUndo = true;
    // Preserve the state we are leaving so redo can restore it.
    m_redo.append(m_shapes);
    if (m_redo.size() > 50) {
        m_redo.removeFirst();
    }
    QList<QVariantMap> previousState = m_history.takeLast();
    setShapes(previousState);
    m_isApplyingUndo = false;
}
```

Add `redo()` immediately after `undo()`:
```cpp
void ShapesModel::redo()
{
    if (m_redo.isEmpty()) {
        return;
    }
    m_isApplyingUndo = true;
    // Current state becomes undoable again.
    m_history.append(m_shapes);
    if (m_history.size() > 50) {
        m_history.removeFirst();
    }
    QList<QVariantMap> nextState = m_redo.takeLast();
    setShapes(nextState);
    m_isApplyingUndo = false;
}
```

**Critical details (do not regress):**
- `setShapes()` already skips `saveHistorySnapshot()` when `m_isApplyingUndo` is true (lines 131–135). Both undo and redo rely on that.
- `redo()` must **not** call `saveHistorySnapshot()` — that would clear `m_redo` before `takeLast()`. Push onto `m_history` directly, same as the undo path pushes onto `m_redo`.
- `beginEdit()` / mutation helpers keep calling `saveHistorySnapshot()` unchanged; the new `m_redo.clear()` inside the snapshot helper is what discards redo on fresh edits.
- Selection-only `updateShape` still does not snapshot (existing `hasRealChanges` guard) — redo stack stays untouched on pure selection flips.

## Acceptance Criteria

1. After any successful `undo()`, `redo()` restores the exact pre-undo `m_shapes` snapshot (types, geometry, style keys).
2. Any new history-producing mutation (`addShape`, non-selection `updateShape`, `removeShape`, `moveShape`, `clear`, `setShapes`, `beginEdit` snapshot) clears the redo stack.
3. `redo()` / `undo()` are no-ops (not crashes) when their stacks are empty.
4. Both stacks remain capped at 50 entries.
5. Overlay: `Ctrl+Shift+Z` (default `action_redo`) invokes redo when the overlay shortcut guard is enabled; rebindable via existing shortcut editor.
6. D-Bus: `redo` slot is callable on `/Overlay` (`ExportAllSlots`).
7. Applet: "Redo" button with `edit-redo` icon sits next to "Undo" and calls `backend.redo()`.
8. All pre-existing undo tests still pass; new redo tests listed above pass.
9. No Excalidraw schema or shape-key changes.
